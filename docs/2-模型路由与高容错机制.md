# 一句话先概括

这个系统不是把 Chat / Embedding / Rerank 直接绑死到某一个模型或某一家供应商上，而是抽象出一套**统一模型路由层**：

1. **先按能力选择候选模型**（Chat / Embedding / Rerank）
2. **再按默认模型、深度思考模型、优先级排序**
3. **结合健康状态过滤不可用模型**
4. **按顺序调用并在失败时自动切到下一个候选**
5. **通过断路器（CLOSED / OPEN / HALF_OPEN）避免雪崩**
6. **对流式 Chat 增加首包探测，避免“连接建立了但一直没内容”这种伪成功**

它的核心目标是：

> **在多模型、多供应商场景下，把“模型选择、失败切换、熔断恢复、流式首包探测”统一收敛为一层基础设施，让上层业务只关注“我要 Chat / Embedding / Rerank”，不需要处理底层故障转移。**

---

# 1. 这套机制主要在解决什么问题？

如果没有模型路由层，业务代码通常会直接写成：

- Chat 永远走某一个模型
- Embedding 永远走某一个接口
- Rerank 永远走某一家服务

这样会有几个明显痛点：

## 1.1 单模型强绑定，任一供应商抖动就会全链路失败
例如：

- Chat 模型接口超时
- Embedding 服务偶发 5xx
- Rerank 服务不稳定

只要绑定的是单点服务，就会直接把上层问答链路打穿。

---

## 1.2 不同能力适合不同模型，不能只靠一个“万能模型”
系统里实际上有三类模型能力：

- Chat
- Embedding
- Rerank

对应配置也分成三组：

- `ai.chat`
- `ai.embedding`
- `ai.rerank`
- `bootstrap/src/main/resources/application.yaml:121-177`

这意味着模型选择本质上不是“选一个模型给全系统用”，而是：

> **不同能力分别配置候选集合，再分别路由。**

---

## 1.3 流式返回比同步返回更复杂，不能把“连接建起来了”当成功
同步 Chat 只要拿到最终响应即可；但流式 Chat 还有一种常见失败场景：

- HTTP / SSE 连接建立成功
- 但迟迟没有首包内容
- 或者无内容直接结束
- 或者启动后很快报错

如果仅以“streamChat 方法返回了 handle”作为成功标准，会把很多**伪成功**误判成成功。

所以这里专门为流式 Chat 增加了**首包探测机制**。

---

## 1.4 故障模型如果持续重试，会拖垮整体延迟
如果某个模型已经明显故障，但系统还在每次请求里都优先去试它，就会出现：

- 每次先超时一次
- 再切换下一个模型
- 整体延迟抖动严重

这也是为什么项目里引入了：

- 失败计数
- 熔断 OPEN
- 半开 HALF_OPEN
- 探活恢复

也就是典型的**断路器模式**。

---

从代码职责上看，可以分成 4 层：

- `ModelSelector`
- `ModelRoutingExecutor`

---

## 第 2 层：候选模型选择层
核心类：

- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/model/ModelSelector.java:40`

职责：

1. 根据能力拿到对应模型组
2. 根据默认模型 / 深度思考模型 / 优先级排序候选
3. 过滤未启用模型
4. 过滤当前不可用模型
5. 组装成 `ModelTarget`

---

## 第 3 层：统一执行与失败切换层
核心类：

- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/model/ModelRoutingExecutor.java:34`

职责：

1. 遍历候选模型
2. 调用具体 client
3. 成功就返回
4. 失败就记录健康状态并 fallback 到下一个
5. 全部失败时抛统一远程异常

同步 Chat / Embedding / Rerank 都走这一层。

---

## 第 4 层：健康状态与熔断层
核心类：

- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/model/ModelHealthStore.java:32`

职责：

1. 记录每个模型的连续失败次数
2. 达到阈值后打开断路器
3. 在熔断窗口内直接拒绝调用
4. 窗口结束后进入 HALF_OPEN，允许少量试探调用
5. 成功则恢复 CLOSED，失败则重新 OPEN

## 4.2 Chat 会区分默认模式与深度思考模式
`ModelSelector.resolveFirstChoiceModel(...)`：

- `ModelSelector.java:63-71`

逻辑是：

1. 如果是 deepThinking 请求
2. 优先使用 `deepThinkingModel`

## 4.3 候选会先做过滤和排序
逻辑在：

- `ModelSelector.java:94-114`

关键规则：

1. 过滤 `enabled=false` 的模型
2. 如果是 deepThinking 模式，只保留 `supportsThinking=true` 的模型
3. 优先把 `firstChoiceModelId` 放到最前面
4. 然后再按 `priority` 从小到大排序
5. 再按 `id` 做稳定排序

也就是说排序优先级是：

> **首选模型 > priority > id**

---

## 4.4 候选在构建 `ModelTarget` 时会先做一次“不可用过滤”
`buildModelTarget(...)` 中：

- `ModelSelector.java:125-139`

如果 `healthStore.isUnavailable(modelId)` 返回 true，就直接不把这个模型放进候选列表：

- `ModelSelector.java:128-130`

这说明系统有两层保护：

1. **候选选择阶段就剔除明显不可用模型**
2. **真正调用前再通过 `allowCall()` 做状态迁移判断**

这能减少无意义的失败尝试。

# 5. 统一执行与故障转移是怎么实现的？

同步能力统一走：

- `ModelRoutingExecutor.executeWithFallback(...)`
- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/model/ModelRoutingExecutor.java:41-78`

## 5.1 执行流程
这个方法的逻辑很清楚：

1. 拿到候选 `targets`
2. 依次遍历每个 target
3. 用 `clientResolver` 解析出具体 provider client
4. 如果 client 缺失，跳过
5. 调 `healthStore.allowCall(target.id())`
6. 真正执行 `caller.call(client, target)`
7. 成功就 `markSuccess()` 并返回
8. 失败就 `markFailure()`，记录日志，切到下一个
9. 全部失败时抛统一 `RemoteException`

对应代码：

- `ModelRoutingExecutor.java:47-77`

---

## 5.2 为什么要把执行器抽象成模板？
`executeWithFallback(...)` 的签名是泛型的：

```java
public <C, T> T executeWithFallback(
        ModelCapability capability,
        List<ModelTarget> targets,
        Function<ModelTarget, C> clientResolver,
        ModelCaller<C, T> caller)
```

见：

- `ModelRoutingExecutor.java:41-45`

这意味着它不关心：

- 你调的是 Chat 还是 Embedding 还是 Rerank
- client 类型是什么
- 返回值是什么

它只关心统一的故障转移流程。

这种设计的好处是：

> **把“业务调用”与“路由容错”彻底解耦。**

上层只需要提供：

- 候选模型列表
- 如何解析 client
- 如何发起具体调用

而失败切换、健康标记、异常统一化都交给执行器处理。

# 7. 高容错的核心：ModelHealthStore 是怎么做断路器的？

核心类：

- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/model/ModelHealthStore.java:32`

这是整个高容错机制的关键。

---

## 7.1 它维护了什么状态？
每个模型 ID 对应一个 `ModelHealth`：

- `consecutiveFailures`
- `openUntil`
- `halfOpenInFlight`
- `state`

见：

- `ModelHealthStore.java:127-138`

状态机是：

- `CLOSED`
- `OPEN`
- `HALF_OPEN`

见：

- `ModelHealthStore.java:141-145`

---

## 7.2 CLOSED：正常状态
刚开始时模型都处于 `CLOSED`。

在这个状态下：

- `allowCall()` 会直接允许请求通过
- 失败会累计 `consecutiveFailures`
- 成功会清零失败计数

---

## 7.3 OPEN：熔断打开
当某模型连续失败次数达到阈值：

- `failureThreshold = 2`
- `application.yaml:114-116`

就会进入 OPEN：

- `ModelHealthStore.java:117-123`

并设置：

- `openUntil = now + openDurationMs`
- `openDurationMs = 30000ms`

此时 `isUnavailable(...)` 会返回 true：

- `ModelHealthStore.java:40-49`

候选选择时会直接把它过滤掉。

---

## 7.4 HALF_OPEN：半开探活
当 OPEN 的窗口过期后，`allowCall(id)` 会把状态从 OPEN 切到 HALF_OPEN：

- `ModelHealthStore.java:62-70`

并且：

- 只允许一个探活请求进入
- `halfOpenInFlight = true`

如果此时还有别的请求并发进入 HALF_OPEN，同一个模型会被拒绝：

- `ModelHealthStore.java:71-78`

这一步很重要，因为它避免了：

> **熔断窗口一结束，大量请求瞬间同时打向一个刚恢复、但尚未验证稳定性的模型。**

---

## 7.5 恢复与重新熔断
### 如果 HALF_OPEN 探活成功
`markSuccess(id)` 会：

- `state = CLOSED`
- `consecutiveFailures = 0`
- `openUntil = 0`
- `halfOpenInFlight = false`

见：

- `ModelHealthStore.java:85-99`

### 如果 HALF_OPEN 探活失败
`markFailure(id)` 会直接重新 OPEN：

- `ModelHealthStore.java:110-116`

这样系统就形成了一个完整闭环：

> **失败累计 → 熔断打开 → 单请求探活 → 成功恢复 / 失败再熔断**

# 8. 为什么流式 Chat 要单独实现，不直接复用执行器？

因为流式调用和同步调用的成功判定标准不同。

同步调用里：

- `caller.call(...)` 返回成功即可认为成功

但流式调用里，仅仅返回 `StreamCancellationHandle` 还不够，必须确认：

- 流已经真正开始输出内容
- 不是卡死
- 不是无内容完成
- 不是启动后立刻报错

因此 `RoutingLLMService.streamChat(...)` 做了单独实现：

- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/RoutingLLMService.java:101-157`

# 9. 流式 Chat 的高容错机制是怎么做的？

## 9.1 基本流程
`streamChat(...)` 的整体逻辑是：

1. 先选 Chat candidates
2. 如果候选为空，直接报错
3. 依次尝试每个 target
4. 解析对应 `ChatClient`
5. `healthStore.allowCall(target.id())`
6. 创建 `ProbeStreamBridge`
7. 调 `client.streamChat(...)`
8. 如果启动异常或 handle 为空，记失败并切下一个
9. 如果 handle 正常，等待首包探测结果
10. 首包成功才 `markSuccess()` 并正式返回 handle
11. 否则取消 handle、记失败、切下一个
12. 所有模型都失败后，通过 callback 通知前端错误

对应代码：

- `RoutingLLMService.java:112-156`

---

## 9.2 首包探测超时时间
常量：

- `FIRST_PACKET_TIMEOUT_SECONDS = 60`
- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/RoutingLLMService.java:49`

也就是说，流式模型在 60 秒内如果拿不到首包结果，就会被视为失败并切下一个候选。

---

## 9.3 ProbeStreamBridge：为什么要加桥接器？
核心类：

- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/ProbeStreamBridge.java:32`

它的职责不是单纯转发流，而是：

1. **先缓存下游回调事件**
2. **等待首包探测结果**
3. **只有确认成功后才 commit 并把缓存事件真正发给下游**

这解决了一个很关键的问题：

> 如果模型很快吐出内容，但我们还没确认“首包成功”，不能直接把内容提前发给客户端；否则一旦后面判定为失败并切模型，前端会看到半截错误输出。

`commit()` 的实现非常简单：

- `ProbeStreamBridge.java:87-95`

```java
private void commit() {
    synchronized (lock) {
        if (committed) {
            return;
        }
        committed = true;
        buffer.forEach(Runnable::run);
    }
}
```

也就是说：

- 成功前：事件先缓冲
- 成功后：一次性释放缓冲并转为直通

---

## 9.4 首包探测如何判定成功？
`LlmFirstPacketProbe.awaitFirstPacket(...)` 实际只是把探测包装成独立 bean，便于 AOP 记录 TTFT：

- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/LlmFirstPacketProbe.java:34-38`

真正等待逻辑在：

- `ProbeStreamBridge.awaitFirstPacket(...)`
- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/ProbeStreamBridge.java:71-85`

逻辑：

1. 阻塞等 probe future 完成
2. 超时则返回 `TIMEOUT`
3. 执行异常则返回 `ERROR`
4. 如果结果是 success，则自动 `commit()`

代码里明确写了：

> `SUCCESS 时自动提交缓冲`

---

## 9.5 流式失败有哪些类型？
在 `RoutingLLMService.buildLastErrorAndLog(...)` 中，至少处理了：

- `ERROR`
- `TIMEOUT`
- `NO_CONTENT`
- default unknown

见：

- `infra-ai/src/main/java/com/nageoffer/ai/ragent/infra/chat/RoutingLLMService.java:182-210`

这比简单的 try-catch 更健壮，因为它明确区分了：

- 启动失败
- 首包超时
- 无内容完成
- 运行中错误

因此流式 Chat 的容错机制明显比同步调用更细。

# 10. 这套机制里有哪些关键设计亮点？

## 10.1 统一路由骨架，能力侧无感知
Chat / Embedding / Rerank 都用：

- `ModelSelector`
- `ModelRoutingExecutor`
- `ModelHealthStore`

业务层不用自己处理：

- 优先级
- 失败切换
- 熔断恢复
- 异常统一化

---

## 10.2 配置驱动，不需要改业务代码就能调整路由策略
新增候选模型、调整优先级、启停模型，只要改：

- `application.yaml`
- `AIModelProperties`

不需要改上层调用方。

这让模型治理从“写死在代码里”变成了“配置化路由”。

---

## 10.3 熔断状态与路由选择联动
不是简单地“失败就切下一个”，而是通过 `ModelHealthStore` 让健康状态参与候选筛选。

这意味着失败模型不会每次请求都被反复命中，从而：

- 降低平均延迟
- 降低失败放大效应
- 避免雪崩

---

## 10.4 HALF_OPEN 只放一个探活请求
这是一个很实用的工程细节。

很多系统有 OPEN / HALF_OPEN，但没有做到**半开期间单飞（single flight）**，结果一恢复就被并发流量打爆。

当前实现通过：

- `halfOpenInFlight`
- `ModelHealthStore.java:67-76`

把 HALF_OPEN 做成“单探针恢复”，工程上更稳。

---

## 10.5 流式 Chat 不是“调用成功”，而是“首包成功”
这一点特别值得在面试里强调。

因为很多系统在流式接口上只判断：

- 方法有没有抛异常
- 是否返回 handle

但这个项目进一步判断：

- 是否真的在合理时间内拿到了第一包有效内容

这使得流式链路的“可用性定义”更接近真实用户体验。

# 11. 这套方案的局限和后续可优化点

为了讲得更完整，也可以顺带提两个当前实现的边界。

## 11.1 fallback 是串行的，不是并行竞速的
当前 `executeWithFallback(...)` 是按候选顺序串行尝试：

- 当前失败才切下一个
- `ModelRoutingExecutor.java:52-70`

优点：

- 简单
- 成本可控
- 易观测

缺点：

- 如果前几个模型都慢失败，尾延迟会变长

后续可演进成：

- hedged requests
- 小窗口并行竞速
- 按历史延迟自适应排序

---

## 11.2 健康状态是进程内内存态
`ModelHealthStore` 用的是：

- `ConcurrentHashMap`
- `ModelHealthStore.java:38`

这意味着它是**单实例内存态**，不会天然跨实例共享。

在单实例或轻量部署下没问题；但如果未来多实例部署，可能需要升级为：

- Redis / 本地 + Redis 双层健康态
- 或接入统一的服务治理层

---

## 11.3 显式指定 modelId 时，fallback 会收窄为单模型
例如：

- `RoutingLLMService.chat(request, modelId)`
- `RoutingEmbeddingService.embed(text, modelId)`

它们都会把候选收缩成：

- `List.of(resolveTarget(modelId))`

见：

- `RoutingLLMService.java:93-98`
- `RoutingEmbeddingService.java:70-75`

这意味着“显式指定模型”更像强制路由，而不是“从指定模型开始 fallback”。

这是合理设计，但面试里可以主动说明这个取舍。

# 14. 面试时可以怎么讲

如果面试官问“模型路由和高容错怎么做的”，你可以这样回答：

> 我们把 Chat、Embedding、Rerank 都抽象成统一的模型路由层。先由 ModelSelector 根据能力、默认模型、深度思考模型和优先级挑选候选，再由 ModelRoutingExecutor 统一执行失败切换；同时用 ModelHealthStore 维护每个模型的健康状态，实现 CLOSED / OPEN / HALF_OPEN 的断路器，连续失败 2 次就熔断 30 秒，恢复时只放一个探活请求。对于流式 Chat，我们没有把“连接建立成功”当作成功，而是额外做了首包探测：只有 60 秒内拿到首包内容才算成功，否则取消并切换下一个候选模型。这样上层业务不需要关心底层 provider 抖动，也能保证模型调用链路整体更稳定。
