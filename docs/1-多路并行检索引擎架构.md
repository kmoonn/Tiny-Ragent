# 一句话先概括

这个系统的核心不是“查一个向量库”，而是一个**分层并行、按置信度选择通道、再统一后处理**的 RAG 检索架构：

1. **先做问题改写与拆分**
2. **再做意图识别**
3. **每个子问题并行构建上下文**
4. **KB 检索内部再走多通道并行**
   - 高精度：意图定向检索
   - 兜底/补充：全局向量检索
5. **多通道结果统一去重、Rerank**
6. **再与 MCP 工具结果合并**
7. **最后构造成 Prompt，流式返回给前端**

这套设计的重点是：**不是单层并行，而是多层并行**。

# 11. MultiChannelRetrievalEngine 的总体结构：两阶段

核心方法：

- `MultiChannelRetrievalEngine.java:64-77`

```java
SearchContext context = buildSearchContext(subIntents, topK);

// 阶段1：多通道并行检索
List<SearchChannelResult> channelResults = executeSearchChannels(context);

// 阶段2：后置处理器链
return executePostProcessors(channelResults, context);
```

也就是说它被设计成标准的两段式：

## 阶段 1：召回（Recall）
多个通道各自独立检索，尽量把候选 Chunk 拉回来

## 阶段 2：统一处理（Merge / Filter / Rerank）
对多通道结果做去重、融合、重排，最终压缩成更高质量结果

这比“一个通道自己做完所有事”更容易扩展。

- `IntentDirectedSearchChannel.java:142-148`

## 16.2 它怎么检索？
核心在：

- `IntentDirectedSearchChannel.java:81-132`

步骤是：

1. 提取 KB 意图
2. 读取 `topKMultiplier`
3. 调 `retrieveByIntents(...)`
4. 返回 `SearchChannelResult`

## 16.3 这里有第三层并行：按意图节点并行检索
`retrieveByIntents(...)` 实际交给：

- `IntentParallelRetriever.executeParallelRetrieval(...)`
- `IntentDirectedSearchChannel.java:153-159`

- `bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/channel/strategy/IntentParallelRetriever.java:35`

- `IntentParallelRetriever.java:52-63`

1. 默认用外部 fallbackTopK
2. 如果意图节点自己配置了 `node.topK`，优先用它

这个设计很聪明，等于：

也就是说，**定向检索不是扫全库，而是直接打到某个意图节点对应的 collection**。

这就是它“高精度”的原因。

# 21. 后处理器 1：去重 DeduplicationPostProcessor

核心逻辑在：

- `bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/postprocessor/DeduplicationPostProcessor.java:57`

做法是：

1. 先按通道优先级排序
2. 用 `LinkedHashMap` 做有序去重
3. 如果重复 chunk 出现，保留分数更高的那个

见：

- `DeduplicationPostProcessor.java:61-87`

这里有个细节值得注意：

- 结果先按通道优先级遍历，所以高优通道有先手优势
- 但如果后面重复 chunk 的分数更高，仍然会被替换

也就是说它不是简单的“高优先级永远覆盖”，而是：

> **优先级决定初始保留顺序，分数决定最终替换。**

这是一种比较平衡的融合策略。

核心逻辑在：

- `bootstrap/src/main/java/com/nageoffer/ai/ragent/rag/core/retrieve/postprocessor/RerankPostProcessor.java:60`

它非常直接：

```java
return rerankService.rerank(
    context.getMainQuestion(),
    chunks,
    context.getTopK()
);
```

见：

- `RerankPostProcessor.java:69-73`

也就是说：

- 前面的通道通常会用 `topKMultiplier` 做**扩召回**
- 最后用 Rerank 对全量候选重新打分并截断成最终 `topK`

这是典型的 RAG 检索设计套路：

> **Recall 尽量多，Precision 靠 rerank 收口。**

---

# 23. 为什么要“多通道 + 后处理链”而不是一个大检索器？

因为两者关注点不同：

## 多通道解决的是“召回覆盖率”
- 意图定向通道：精准
- 全局向量通道：兜底和补充

## 后处理链解决的是“结果质量”
- 去重
- 融合
- 重排
- 控制最终输出数量

如果把这两类逻辑揉在一起，会导致：

- 通道难扩展
- 合并逻辑混乱
- 可观测性差

而现在这种设计符合经典模式：

- **SearchChannel = Recall 插槽**
- **PostProcessor = Ranking / Fusion 插槽**

# 24. 检索结果如何回到 RetrievalEngine？

`retrieveAndRerank(...)` 在拿到多通道最终 chunk 后，会做一个额外处理：

- `RetrievalEngine.java:208-225`

它构建 `intentChunks`：

```java
for (NodeScore ns : kbIntents) {
    intentChunks.put(ns.getNode().getId(), chunks);
}
```

见：

- `RetrievalEngine.java:211-218`

这里代码注释明确写了一个现实问题：

> 多通道检索返回的 chunks 无法精确对应到某个意图节点  
> 所以把所有 chunks 分配给每个意图节点

这说明当前架构里：

- 通道结果能知道“来自哪个通道”
- 但不能精确追溯“来自哪个具体 intent node”

因此在 prompt formatting 阶段做了一个**近似映射**。

然后调用：

- `contextFormatter.formatKbContext(kbIntents, intentChunks, topK)`
- `RetrievalEngine.java:224`

把检索结果变成可注入 Prompt 的 KB 文本。

# 25. MCP 结果和 KB 结果如何合并？

还是在 `buildSubQuestionContext(...)`：

MCP 工具调用得到 `mcpContext`

- `SubQuestionContext(question, kbContext, mcpContext, intentChunks)`

# 28. 这套多路并行检索架构，设计上最值得注意的 7 个点

## 1）多层并行，而不是单点并行
这套系统至少有 4 层并行：

1. **子问题意图识别并行**  
   `IntentResolver.java:56-67`

2. **子问题上下文构建并行**  
   `RetrievalEngine.java:85-99`

3. **检索通道并行**  
   `MultiChannelRetrievalEngine.java:96-109`

4. **通道内部目标并行（按 intent / collection）**  
   `AbstractParallelRetriever.java:68-75`

这能显著降低整体尾延迟。

---

## 2）高精度通道 + 兜底通道组合
- `IntentDirectedSearchChannel`：精确命中
- `VectorGlobalSearchChannel`：低置信度兜底 / 补召回

这是“精准率”和“召回率”的平衡。

---

## 3）置信度驱动的通道启用
全局检索不是一直开，而是由意图置信度决定：

- 没意图 → 开
- 低置信度 → 开
- 单一中置信度 → 补开

这样避免高置信度场景下无意义扫全库。

---

## 4）先扩召回，再 Rerank 收敛
通道里通过 `topKMultiplier` 拉大候选集，
最后用 `RerankPostProcessor` 压缩到最终 `topK`。

这是 RAG 里非常成熟的模式。

---

## 5）接口解耦清晰
- `SearchChannel`：召回策略抽象
- `SearchResultPostProcessor`：后处理抽象
- `RetrieverService`：底层向量检索抽象

所以新增通道、替换向量库、增加 rerank 都不需要推翻主流程。

---

## 6）每层都做了降级容错
例如：

- 子问题意图分类失败 → 降为空意图  
  `IntentResolver.java:59-64`

- 通道执行失败 → 返回空结果  
  `MultiChannelRetrievalEngine.java:99-105`

- 单个目标检索失败 → 记日志但不中断整体  
  `IntentParallelRetriever.java:77-80`  
  `CollectionParallelRetriever.java:53-56`

- 后处理器失败 → 跳过当前处理器继续链路  
  `MultiChannelRetrievalEngine.java:174-189`

这说明设计目标不是“局部必须成功”，而是“整体尽量有结果”。

---

## 7）可观测性做得比较到位
各层都有统计日志：

- 通道级统计  
  `MultiChannelRetrievalEngine.java:121-143`

- 通道内部目标统计  
  `AbstractParallelRetriever.java:78-97`

- 后处理器前后数量变化  
  `MultiChannelRetrievalEngine.java:173-193`

这对线上调优很重要。
